# Lessons

- When adding editor-event listeners in the extension host, always guard refresh callbacks with `try/catch` or an error-reporting boundary before shipping. Unit tests can pass while runtime-only editor events still throw and surface as `FAILED to handle event`.
- When forking a VS Code extension for side-by-side installation, changing only `publisher` is not enough. Commands, custom editor view types, webview types, and extension-owned context keys also need unique fork-specific namespaces or activation can fail due to duplicate registrations.
- When a user says a packaged VSIX fix still behaves the same, do not assume the previous root-cause theory was sufficient. Re-check activation events, menu `when` clauses, and whether the build version changed enough for the new package to be unmistakably installed.
- When a visibility or highlighting fix still fails after a manifest tweak, verify the real document language IDs in the target workflow before iterating again. HOI4 companion extensions may use IDs different from `plaintext`, `hoi4`, or `paradox`, and brittle menu gates hide the real issue.
- When the user provides a concrete VS Code log file for an unresolved runtime issue, stop iterating on hypotheses and anchor the next fix to the logged error first.
- When a runtime error points at a utility abstraction like lodash `chain`, prefer replacing that abstraction on the hot path with a small native implementation instead of trying to preserve a brittle shimmed chain API.
- When broadening menu visibility as a fallback, keep the fallback entry mutually exclusive with the context-driven entry or VS Code will render duplicate toolbar buttons.
- When the user reports one surface is fixed but another still fails, narrow the next pass to that subsystem only instead of reopening the whole activation theory.
- When the user offers a concrete edge-case hypothesis like a spaced localisation filename, turn it into a direct regression test instead of only broadening the theory in prose.
- Do not bump the extension version for every intermediate debugging pass. In this repo, keep version changes for intentional release batches or user-facing install points, and group multiple small fixes under one planned version when possible.
- When a highlighting feature feels slow, inspect its editor event subscriptions before touching parsing logic. Recomputing decorations on `visibleRanges` or every refresh is usually the hot path in VS Code, so cache by document version and skip unchanged editor applications first.
- When exposing a new setting in the VS Code Settings UI, do not stop at the manifest. Verify that the runtime path reads the setting dynamically or reloads affected surfaces on configuration changes, otherwise the toggle appears to work but has no visible effect.
- When adding experimental editor UI to a preview webview, do not default to a floating inspector. In this repo, overlapping popup controls are worse than missing controls, so keep layout-edit affordances inline and reserve drag for direct position changes.
- When the user asks for a minimal editing affordance, do not leave backup controls or status chrome behind. In this preview UI, one clear `Edit` toggle plus direct drag behavior is better than a half-simplified toolbar with extra apply/status controls.
- When simplifying the focus layout editor to drag-only interaction, do not rebuild the preview on every mousemove. Keep the live drag visual local in the webview and commit the text edit once on mouseup so pointer capture stays reliable.
- When direct manipulation is added to a preview that already has full-surface pan or scroll capture, explicitly disable that background interaction layer during edit mode. Otherwise drag-start can fail even when the new drag code itself looks correct.
- When correcting preview interaction regressions, do not leave half-migrated input models in place. In this repo, either keep the original pan path intact or fully replace it, and make drag operate on the actual clicked element rather than a fresh global query.
- When a preview needs both normal panning and edit-mode dragging, do not rely on a shared full-surface drag layer plus ad-hoc interlocks. In this focus preview, it is more reliable to own both input modes in the preview script and switch behavior explicitly.
- When preview editing depends on direct node dragging, audit fullscreen interaction layers like `#dragger` before changing drag math. A pan or zoom overlay can steal pointer input even when the drag code itself is correct.
- When reintroducing focus preview dragging, do not trust `event.target` alone. In this webview, fullscreen layers like `#dragger` can still win the initial hit test, so drag start should fall back to `elementFromPoint()` with the overlay temporarily disabled.
- When wiring preview interactions to rendered DOM attributes, verify the selector against the exact emitted HTML before debugging event math. In this focus preview, `data-focus-id` lived on a different element than the `.focus` class, so `.focus[data-focus-id]` bound nothing at runtime.
- When a preview edit writes back through document versions, do not hard-fail solely because the webview version is stale. In this repo, preview refresh lag after a successful edit can make the next drag race the HTML refresh, so the host path should remain resilient and the webview should refresh its local version state.
- When edit mode needs both drag editing and navigator clicks, do not suppress all clicks wholesale. In this focus preview, only a real drag should cancel the follow-up click; a simple click in edit mode should still navigate to the focus definition.
- When a preview already applies an optimistic local update for its own edit, do not let the normal debounced document-change rerender replay the whole webview a moment later. In this focus preview, self-authored coordinate edits should short-circuit the expensive full rebuild path.
- When suppressing a delayed host rerender for performance, make sure an immediate replacement render still exists. In this focus preview, skipping the host refresh without triggering a local rebuild left the DOM stale until some later UI action forced a redraw.
- When a user asks for feature implementation that is intended to ship, do not leave the extension version unchanged. In this repo, bump `package.json`, `package-lock.json`, and `CHANGELOG.md` together before calling the work done.
- When shipping a user-requested editor feature in this extension, do not leave the UI hidden behind a default-off setting unless the user explicitly wants a gated rollout.
- When the user asks to remove a preview feature entirely, remove its runtime wiring, settings, migrations, tests, and release-note residue together instead of merely hiding the UI or leaving stale compatibility code behind.
- When a user gives an exact template body for generated HOI4 content, do not keep the earlier minimal placeholder template. Update the generator and its regression tests to match the requested scaffold exactly, including repeated placeholder tokens and blank lines.
- When a preview screenshot shows only the continuous-focus overlay and no nodes, audit visual stacking and empty-state filtering before blaming parsing. In this focus preview, an absolute helper overlay can hide the grid, and stale condition state can make a valid tree render zero items.
- When blank-canvas editing actions do not fire in the preview, do not trust `dblclick` `event.target` alone. In this webview, positioned layers and empty container gaps can report misleading targets, so creation hit-tests should recheck the real pointer location with `elementFromPoint()`.
- When the focus preview background visually extends beyond the actual gridbox DOM bounds, do not require blank-space clicks to land inside a specific container box. Exclude known interactive regions instead, then allow the rest of the canvas to create.
- When generated placeholder ids can remain unchanged across multiple create operations, do not reuse the same placeholder verbatim. Pick a unique id like `TAG_FOCUS_ID_2` from the current file state so later edit paths do not fail with ambiguous focus ids.
- When converting preview pixel positions into HOI4 grid coordinates for create-on-click flows, do not use nearest-cell rounding. Use the containing cell instead, or clicks on the right half of a slot will create in the adjacent column.
- When changing a focus's `relative_position_id` from the preview, do not write only the new parent id. Recalculate and save the child focus's local `x` and `y` against that new parent so the rendered position does not jump.
- When a user treats several fast follow-up fixes as one logical release, do not keep bumping the extension version on every micro-fix. Consolidate `package.json`, `package-lock.json`, and `CHANGELOG.md` into a single release number unless the user explicitly wants separate published versions.
- When a preview toolbar needs pixel-clean alignment, do not rely on whatever baseline the default inline elements happen to use. Group related controls in an explicit flex row and center-align the button, label, and input together.
- When a user gives an exact toolbar control order, do not preserve the previous grouping out of convenience. Rebuild the toolbar groups so the requested order and alignment are both explicit in the markup.
- When a preview has multiple condition pickers that conceptually control the same visibility logic, do not keep parallel state flows after the user asks to unify them. Remove the duplicate control and make the surviving dropdown drive both the UI options and the rendered result.
- When a focus preview toolbar starts to drift out of alignment, do not keep compressing more controls into one row. In this repo, split crowded preview toolbars into explicit rows and update the content offset together with the toolbar height.
- When toolbar visibility is toggled in the preview webview, do not switch a flex control group back to `display: block`. In this repo, runtime show/hide code must preserve the intended layout mode or labels and inputs will unexpectedly stack.
- When blank-space creation depends on scrolling the preview canvas, do not size the canvas to stop exactly at the last rendered node. Leave explicit edit buffer space beyond the current content so users can keep creating focuses outward.
- When a focus preview still relies on the shared `#dragger` pan layer, do not assume every visible blank-canvas hit will reach it. If content containers can sit above the overlay, add a blank-space pan fallback in the preview script rather than treating the drag miss as user error.
- When adding a blank-canvas pan fallback for the preview, do not invent a second independent scroll implementation. Reuse the same pan session logic as `#dragger`, or the fallback can feel rougher than the original drag behavior.
- When a user asks for a toolbar button to visually match an existing icon control, do not stop at swapping the glyph. Rebuild the row layout so the icon lands in the requested edge position and update the active-state styling to suit an icon button.
- When a user corrects a toolbar position request with a more specific anchor like "right after Search", do not keep generic auto-margin alignment in place. Remove the edge-push rule and anchor the control to the named neighbor explicitly.
- When adding a custom webview context menu, do not treat host-side edit wiring and unit tests as sufficient proof that the action works. Protect the menu's own pointer events from document-level click handlers and carry the action payload on the menu DOM itself so runtime delete clicks cannot lose their target id.
- When `Edit` mode disables the shared preview pan layer, do not forget to re-enable an equivalent blank-canvas pan path inside the preview script. In this focus preview, direct-node drag and blank-space pan must both remain available while editing.
- When focus preview performance regresses on large trees, do not start with cosmetic micro-tweaks. Cut repeated DOM-wide scans and linear lookups from the webview hot path first, then trim host-side array churn only after the interactive path is cheap.
- When the preview still feels slow after webview hot-path cleanup, do not stop at renderer-side optimizations. Check the preview manager's document-change debounce next, because a 1000ms host-side delay can dominate perceived responsiveness regardless of in-webview speed.
- When the focus preview still feels slow after cutting hot-path work and debounce, do not keep replacing the entire webview HTML on every same-document edit. In this repo, split render payload generation from shell HTML and refresh the already-loaded focustree webview in place whenever the toolbar structure has not changed.
- When the user extends a newly shipped editor affordance to a parallel HOI4 definition family such as ideologies after an initial country-only pass, treat that as a scope correction: update the same feature batch, widen file targeting/tests together, and avoid leaving the neighboring data family unsupported.
- When a user re-applies an existing preview relation, do not treat it as a harmless no-op. In this focus preview, repeating the same prerequisite or mutually exclusive link should toggle it off, and prerequisite unlink must preserve the child's rendered position while clearing `relative_position_id`.
- When a user says a focus relation is conceptually mutual, do not only write the clicked source side. In this preview, `mutually_exclusive` edits must add and remove the relation on both focuses so the generated script stays symmetric.
- When leaving blank canvas around the focus preview for editing, do not stop after adding bottom-only space. Expand the effective canvas origin and size so left, top, right, and bottom all keep usable edit buffer.
- When adding marquee multi-select to the focus preview, do not steal the plain blank-space drag gesture if edit-mode pan is already expected. In this repo, keep plain drag for pan and put box selection on an explicit modifier such as `Shift + left drag`.
- When a GitHub Actions `test-ui` failure log explicitly says the development extension cannot find `dist/extension.js`, do not chase the downstream `command not found` or timeout symptoms first. In this repo, `compile-ts` only builds `out/`, so UI tests must build the webpack extension bundle before launching `vscode-test`.
- When a CI smoke test log shows preview telemetry and loader activity before the assertion times out, do not assume the preview command failed. In this repo, webview smoke tests should avoid relying only on `TabInputWebview instanceof` and short waits; prefer a more stable visible-title fallback plus a more realistic timeout.
- When a focus preview screenshot shows only the shell controls and helper overlays while the actual focuses are missing, do not stop at asset-loading theories. In this repo, stale restored webview state like a negative `selectedFocusTreeIndex` can abort the first tree render for single-tree files, so clamp restored indexes before using them and keep a regression test for that recovery path.
- When a user says the focus preview has accreted unnecessary UI, do not keep layering new surfaces on top of the old ones. In this repo, remove the whole feature slice - shell markup, webview state, host messages, and dedicated tests - and re-run tests after a clean build so stale `out/` artifacts do not hide dead feature paths.
- When a user says multi-selected focus links should land in one prerequisite block, do not reuse single-parent overlap toggles. In this preview, carry the whole selected parent set through the message path, merge it into one prerequisite field, and only toggle off when the exact grouped set is re-applied with the same anchor `relative_position_id`.
- When a user asks to roll Conditions back but keep presets, do not keep the preset logic welded into the condition-selection core. Restore the original global condition flow first, then add presets back as a thin per-tree layer that drives the existing dropdown instead of redefining it.
- When a user says the `relative_position_id` anchor for a grouped link should be the top-most focus in that branch section, do not keep using the clicked parent as the anchor out of convenience. Compute the anchor from the selected parent set using rendered positions, and keep that rule in a pure helper with a regression test.
- When a user says the `Conditions` list itself is wrong, do not stop at webview selection state. In this preview, inspect the schema-side `conditionExprs` collection path first and keep it limited to the intended source set such as `allow_branch` triggers.
- When a user asks to remove a focus hover tooltip but keep hover guidance, do not delete the whole hover state. In this preview, remove the overlay surface and keep the guidance by highlighting directly related prerequisite, child, and mutually exclusive focuses through the existing node outline path.
- When a user asks for hover highlighting to feel like the pre-rollback relation view, do not settle for outline-only emphasis. In this preview, restore stronger relation feedback by dimming unrelated focus nodes and relation lines while preserving the tooltip-free hover state.
- When a toolbar helper depends on another control semantically, do not gate it only on the broad feature flag. In this preview, `Preset` should only be shown when the current tree actually exposes `Conditions` options.
- When edit-mode drag or marquee interactions feel like they randomly cancel in the focus preview, do not keep relying on loose document mouse events. In this repo, use explicit pointer ownership with pointer capture, and remember that `Continuous focuses` must have pointer events enabled while editable or it cannot be dragged at all.
- When focus preview icons fail to load for mod-defined `GFX_*` names, do not assume the optional GFX index or default `goals.gfx` fallback is enough. In this repo, resolve icon containers by name first, then fall back to a targeted `interface/*.gfx` scan so workspace-local mod sprites still load when the index misses them.
- When adding a fallback scan over HOI4 `.gfx` files, do not assume every file parses cleanly as a sprite container. In this repo, skip unreadable or unparsable `.gfx` files during icon fallback resolution so one malformed GUI file cannot blank the whole focus preview.
- When a user asks to remove a hover-only visual in the focus preview, do not weaken the whole hover guidance path. In this repo, remove just the hover-specific outline treatment and keep relation dimming/highlighting intact unless the user asks to change those too.
- When a focus preview context-menu action should respect a multi-selection, do not reuse the clicked single-focus path unchanged. In this repo, if the clicked focus is part of the current selection, carry the whole selected id set through the message and apply the writeback as one grouped edit.
